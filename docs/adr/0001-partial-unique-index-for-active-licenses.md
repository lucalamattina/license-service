# Partial unique index for "at most one active license per (user, product)"

The duplicate-license issuance policy is a check-then-act sequence (read the user's existing active license for the product, compare expires_at, revoke-and-replace or reject). Two concurrent issuances for the same (user_id, product_id) can both pass the check and both insert, leaving the user with two Active licenses for the same product — a silent violation of the core domain invariant.

We considered two alternatives: pessimistic locking via `SELECT ... FOR UPDATE` (or a Postgres advisory lock keyed on `hash(user_id, product_id)`), and serializable transaction isolation with retry on conflict. Both work, but both push correctness into application code. The advisory-lock approach in particular relies on every issuance path remembering to take the lock; a future code change that issues a license without going through the canonical path silently breaks the invariant.

We chose a partial unique index on `(user_id, product_id) WHERE status = 'active'`. The invariant is now enforced by the schema itself: it is structurally impossible for two Active licenses for the same (user, product) to coexist, regardless of what the application code does. Issuance runs in a transaction that revokes the existing Active license before inserting the new one; a racing concurrent insert that slips past the application-level check fails on the index and is mapped to 409 `duplicate_active_license`. Constraint violation becomes a second line of defence, not the primary mechanism.

This decision is hard to reverse — changing a uniqueness constraint on a populated `licenses` table requires either a brief write-lock or an `INDEX CONCURRENTLY` dance with downtime tolerance — which is why it's worth recording. Future code that needs to bulk-revoke-and-reissue (e.g. a migration that re-grants entitlements) must do so inside a single transaction in the same revoke-then-insert order, or it will trip the index.
